ここまでで一覧、新規作成、編集、削除の各ページがそろいました。本章では、ユーザーの操作を一連の流れとして確認し、よくあるエラーを洗い出します。最小限の CRUD アプリとして完成したかをチェックする区切りの章です。

## この章で学ぶこと

| 項目 | ポイント |
| --- | --- |
| 操作フローの確認 | 一覧 → 新規作成 → 編集 → 削除の流れを追体験する |
| チェックリスト | ルーティングや DI、マイグレーションの漏れを確認する |
| デバッグのヒント | ブラウザのエラーやログの見方を押さえる |

## 一連の操作フロー

1. 一覧ページ `/books` を開き、登録済みの書籍を確認する
2. 「新規作成」から `/books/create` に移動し、タイトル・著者・メモ・読了を入力して保存する
3. 一覧に戻り、作成した書籍が表示されることを確認する
4. 編集ボタンまたは行クリックで `/books/edit/{id}` に移動し、項目を更新して保存する
5. 同じページで削除ボタンを押し、確認ダイアログで承認すると一覧から消えることを確認する

ユーザー視点で操作を通すことで、ナビゲーションの導線やメッセージ表示の抜けを見つけやすくなります。

## 動かないときのチェックリスト

- ルートが間違っていないか（`/books`、`/books/create`、`/books/edit/{id}`）
- `Program.cs` で `AddDbContext` を登録しているか
- SQLite の接続文字列が正しく、マイグレーションが適用されているか
- DI の注入忘れがないか（`@inject`、`[Inject]` の記載）
- `EditForm` の `OnValidSubmit` が設定されているか

## デバッグのヒント

- ブラウザの開発者ツールでコンソールエラーを確認する
- サーバーのログ（ターミナル出力）で例外メッセージを確認する
- 例外スタックトレースに出てくるコンポーネント名や行番号を頼りに該当箇所を開く

## ひとまずの完成を宣言する

この時点で、書籍管理アプリは CRUD の最小機能を備えています。ここから先は、責務分離や検索・並び替えといった改善に取り組み、より実用的な構成に育てていきます。次章では、サービス層を導入して UI とデータアクセスを整理します。
