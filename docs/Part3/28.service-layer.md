---
title: "サービス層で責務分離しよう"
---

ここからは UI とデータアクセスの責務を分離するため、サービス層を導入します。コンポーネントから DbContext への直接依存をなくし、テストしやすく変更に強い構成へと進めます。

ゴールは、`IBookService` と `BookService` を用意し、一覧・詳細・作成・編集・削除の各処理をサービス経由に切り替えることです。

## この章で学ぶこと

| 項目 | ポイント |
| --- | --- |
| サービス層の意義 | UI とデータロジックを分離し、再利用性とテスト容易性を高める |
| インターフェース設計 | `IBookService` のメソッド構成を決める |
| 実装イメージ | `BookService` が DbContext をどのように利用するかを確認する |
| コンポーネント側の変更 | DbContext 直接参照からサービス注入に置き換える |

## なぜサービス層を導入するのか

- UI ロジックとデータアクセスが混在すると、画面数が増えたときに重複が増える
- テストでデータアクセスをモックしづらくなる
- データソースの変更（SQLite から Azure SQL など）に影響されやすい

サービス層でデータ操作を一本化することで、UI は「どのデータを表示するか」に集中できます。

## IBookService の設計

最低限、次のメソッドを用意します。

- `Task<List<Book>> GetListAsync(BookSearchCondition? condition = null);`
- `Task<Book?> GetByIdAsync(int id);`
- `Task<Book> CreateAsync(Book book);`
- `Task UpdateAsync(Book book);`
- `Task DeleteAsync(int id);`

検索条件を渡せるようにしておくと、後章で検索・並び替えを追加するときに拡張しやすくなります。

## BookService の実装イメージ

`BookService` は AppDbContext を受け取り、検索や更新のロジックをまとめます。例として新規作成の処理は次のようになります。

```csharp
public async Task<Book> CreateAsync(Book book)
{
    book.CreatedAt = DateTime.Now;
    _db.Books.Add(book);
    await _db.SaveChangesAsync();
    return book;
}
```

更新や削除では、まず対象の存在を確認し、見つからない場合は null を返すか例外を投げる方針を決めておきます。

## コンポーネントでサービスを使う

これまで `@inject AppDbContext` としていた箇所を `@inject IBookService BookService` に置き換えます。例えば一覧ページでは次のように取得します。

```razor
protected override async Task OnInitializedAsync()
{
    books = await BookService.GetListAsync();
}
```

コンポーネントがデータ取得の詳細を知らなくてもよくなり、UI の責務が明確になります。サービス層を導入した状態を基点に、次章で検索・フィルタ・並び替えを追加していきます。
