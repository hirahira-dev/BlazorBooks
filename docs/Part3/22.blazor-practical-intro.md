Blazor 本の基礎編で学んだコンポーネント、データバインド、イベント処理を踏まえ、ここからは実践的な書籍管理アプリを題材にステップを進めていきます。双方向バインドやイベントハンドリングはフォーム入力に、DI やライフサイクルはデータ取得の流れに活きるため、Part3 で「どの場面で基礎が効いてくるのか」を意識しやすくなります。全体の流れをあらかじめ把握し、ゴールを意識しながら読み進められるよう、この章ではアプリの概要と到達点を整理します。

本章の目的は、実践編（Part3）で作るアプリの全体像を示し、学習の見取り図を持ったうえで各章に臨めるようにすることです。開発の手順を一通り体験し、実務に近いレイヤ構成の意図を理解することがゴールになります。

## 書籍管理アプリの全体像

書籍管理アプリは、読んだ本やこれから読みたい本を記録するシンプルな CRUD アプリです。業務アプリの多くは CRUD が中心で、データの読み書きに伴う設計判断を学ぶ題材として適しています。主な画面は次のとおりです。

- 書籍一覧ページ：登録済みの書籍をテーブルで表示し、検索や並び替えを行う
- 新規登録ページ：書籍情報を入力して保存する
- 編集ページ：既存の書籍を編集・削除する

実装のベースは Blazor Web App（Server インタラクティブ）と Entity Framework Core（SQLite）です。まず最小限の CRUD を成立させ、その後検索・フィルタ・並び替えを追加し、コードを整理していきます。

## Part3 で取り組む機能

本パートでは、次の順序で機能を育てていきます。

1. プロジェクトの準備とデータモデル作成（Book エンティティ、DbContext、マイグレーション）
2. 一覧表示（READ）
3. 新規作成（CREATE）
4. 編集・削除（UPDATE/DELETE）
5. CRUD 全体の動作確認
6. サービス層導入による責務分離
7. 一覧の検索・フィルタ・並び替え改善
8. 全体リファクタリングとフォーム共通化

章が進むほど UI とデータアクセスの分離が進み、実務的な設計に近づきます。

## 実務とのつながり

本パートで採用するレイヤ構成（UI とサービス層の分離、DbContext の抽象化）は、業務システムでも広く使われる考え方です。単純な CRUD であっても、後から検索条件が増えたり、データソースが変わったりすることがあります。最初から責務を分けておくことで、機能追加やテストを行いやすくなります。

ここで得た設計感覚は、社内ツールやマスタ管理画面など、実務で頻出するシナリオにそのまま応用できます。次章から実装を進め、最終章で全体を振り返りながら Part4（クラウド・認証）へとつなげていきましょう。
