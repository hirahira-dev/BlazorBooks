title: "アプリ全体をリファクタリングしよう"
---

機能がそろったところで、コードを整理し保守しやすい構成に整えます。本章ではフォルダ構成や共通コンポーネント化、命名の見直しといった入門レベルのリファクタリングに取り組みます。

ゴールは、Pages と Components、Services の責務をはっきり分け、BookForm などの共通部品で重複を減らすことです。

## この章で学ぶこと

| 項目 | ポイント |
| --- | --- |
| フォルダ構成の整理 | Pages/Books、Components/Books、Services などに役割を分ける |
| フォーム共通化 | BookForm コンポーネントで新規作成・編集フォームを共通化する考え方 |
| 命名と責務 | メソッド名やコンポーネント名を役割に沿って見直す |

## フォルダ構成の見直し

- Pages/Books：ルーティングを持つページコンポーネント（一覧・新規作成・編集）
- Components/Books：BookListRow、BookForm などの部品化したコンポーネント
- Services：IBookService、BookService と関連するモデル

レイヤごとに配置を分けることで、ファイルの所在地が直感的になり、新しい開発者も迷いにくくなります。

## BookForm によるフォーム共通化

新規作成と編集で共通する入力項目を BookForm に集約します。親コンポーネントはモデルとコールバックを渡すだけにし、ボタンやラベルの違いをパラメーターで切り替える設計がシンプルです。

- `BookForm` パラメーター例：`Book Model`、`EventCallback OnValidSubmit`、`EventCallback OnDelete`、`bool IsEdit`
- バリデーションや入力コンポーネントは共通化して、表示テキストのみ親が制御する

これにより、フォームの変更（項目追加やスタイル変更）を一箇所で済ませられます。

## 命名や責務の確認ポイント

- メソッドは動詞から始め、目的が分かる名前にする（例：`LoadAsync`、`SaveAsync`）
- コンポーネントは表示責務に集中させ、データ取得や整形はサービス層へ寄せる
- DI で受け取るサービスやコンテキストは、必要最低限にする
